建造者模式[Builder Pattern]

2019-09-03
Java 设计模式

1. 为什么要有建造者模式?

靠靠靠，觉得真无聊？请直接往下翻到代码和代码中的注释，看完后在来看学术性的东西吧….

现实中，大兄弟去买汽车，他要关心汽车用的是蓝翔的焊接技术还是绿翔的技术么？不用。他关心的是得到一部完好的汽车；软件开发中也有类似的问题，一个炒鸡复杂的对象，他又N多个属性和方法，但是这些属性和方法不用每个人都去关心，我们关心的是得到相应的对象[建造者模式可以构建不同对象]，然后使用它；

2. 建造者模式是什么？

  • 将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。
  • 建造者模式是一步一步创建一个复杂的对象，它允许用户只通过指定复杂对象的类型和内容就可以构建它们，用户不需要知道内部的具体构建细节。建造者模式属于对象创建型模式。根据中文翻译的不同，建造者模式又可以称为生成器模式。

3.建造者模式包括什么?

  • Builder：抽象建造者
  • ConcreteBuilder：具体建造者
  • Director：指挥者
  • Product：产品角色

4. 举个栗子

例子往往可以帮我们很好的理解

例子： 金拱门(“小的觉得麦当劳还好听些”)买一份快餐，假设由主食(汉堡/鸡肉卷等)和饮料(可乐/橙汁等)组成，根据顾客不同的要求，服务员需要一步一步的构建出不同的套餐，然后返回将套餐返回给顾客；那么我们是顾客，我们点一分快快餐，也就是Product；服务员（Director）知道我们要的快餐类型后交由具体的厨师制作（ConcreteBuilder）;最后将一份快餐返还给顾客手中。下面的代码大概就是这样一个流程；

先来个UML图压压惊，有个整体的了解 

//具体套餐【Product：产品角色】
class Meal{
    String food;
    String drink;

    public void setDrink(String drink) {
        this.drink = drink;
    }
    public void setFood(String food) {
        this.food = food;
    }

    @Override
    public String toString() {
        return food+drink;
    }
}
//【Builder：抽象建造者】
abstract class Builder{
    Meal meal = new Meal();
    //step1 选择主食
    abstract void buildFood();

    //step2 选择饮料
    abstract void buildDrink();

    public Meal getMeal() {
        return meal;
    }
}

//具体的A套餐构建者【ConcreteBuilder：具体建造者】
class ConcreteBuilderA extends Builder{

    void buildFood(){
        meal.setFood("foodA");
    }

    void buildDrink(){
        meal.setDrink("drinkA");
    }

}
//具体的B套餐构建者【ConcreteBuilder：具体建造者】
class ConcreteBuilderB extends Builder{

    void buildFood(){
        meal.setFood("foodB");
    }

    void buildDrink(){
        meal.setDrink("drinkB");
    }

}

//服务员【Director：指挥者】
class Director{
    void construct(Builder builder){
        builder.buildFood();
        builder.buildDrink();
    }
}

//具体实施调用
public class Main{

    public static void main(String[] args) {
        Director director = new Director();
        Builder builder = new ConcreteBuilderA();
        director.construct(builder);
        System.out.println(builder.getMeal());

    }

}

5. 分析

  • 建造者模式的指挥者类Director的作用两个：

  1. 一方面它隔离了客户与生产过程；
  2. 另一方面它负责控制产品的生成过程。指挥者针对抽象建造者编程，客户端只需要知道具体建造者的类型，即可通过指挥者类调用建造者的相关方法，返回一个完整的产品对象

  • 优点：

  1. 在建造者模式中， 客户端不必知道产品内部组成的细节，将产品本身与产品的创建过程解耦，使得相同的创建过程可以创建不同的产品对象。
  2. 每一个具体建造者都相对独立，而与其他的具体建造者无关，因此可以很方便地替换具体建造者或增加新的具体建造者， 用户使用不同的具体建造者即可得到不同的产品对象 。
  3. 可以更加精细地控制产品的创建过程 。将复杂产品的创建步骤分解在不同的方法中，使得创建过程更加清晰，也更方便使用程序来控制创建过程。
  4. 增加新的具体建造者无须修改原有类库的代码，指挥者类针对抽象建造者类编程，系统扩展方便，符合“开闭原则”。

  • 缺点

  1. 建造者模式所创建的产品一般具有较多的共同点，其组成部分相似，如果产品之间的差异性很大，则不适合使用建造者模式，因此其使用范围受到一定的限制。
  2. 如果产品的内部变化复杂，可能会导致需要定义很多具体建造者类来实现这种变化，导致系统变得很庞大。

  • 适用环境

  1. 需要生成的产品对象有复杂的内部结构，这些产品对象通常包含多个成员属性。
  2. 需要生成的产品对象的属性相互依赖，需要指定其生成顺序。
  3. 对象的创建过程独立于创建该对象的类。在建造者模式中引入了指挥者类，将创建过程封装在指挥者类中，而不在建造者类中。
  4. 隔离复杂对象的创建和使用，并使得相同的创建过程可以创建不同的产品。
